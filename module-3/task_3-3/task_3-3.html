1. [What is the difference between invoking and creating a function?]

    2. [Å opprette (definere) en funksjon betyr å skrive selve funksjonen — du lager en blokk med kode som kan gjenbrukes senere. Å kalle (invokere) en funksjon betyr å utføre den koden ved å bruke funksjonsnavnet etterfulgt av parenteser.]

  2. [Explain the syntax regarding semicolons and code blocks.]

    3. [semikolon avslutter statements (bruk dem konsekvent), og {} definerer blokker og dermed variabellivslengde (scope).]

  3. [What happens when we try to store values from a function when it doesn't return any?]

    4. [Hvis en funksjon ikke returnerer noe, vil et kall til den gi verdien undefined. Dersom du gjør f.eks. const x = fn(); blir x undefined, og å prøve å bruke egenskaper eller metoder på undefined gir en runtime-feil (TypeError).]

  4. [How do we separate parameters from each other when creating a function?]

    5. [Parametre skilles med komma inni parentesene etter funksjonsnavnet. Eksempel: function minFunksjon(a, b, c) { /* ... */ }.]

  5. [Explain how parameter names are related during the invocation and creation of a function]

    6. [Parameterne i funksjonsdefinisjonen er lokale plassholdere (lokale variabler) som får verdier fra argumentene ved kall. Argumentene matches etter posisjon: første argument -> første parameter osv. Navnene må ikke være like variablene i koden som kaller funksjonen — de fungerer bare lokalt inne i funksjonen. Primitiver kopieres, objekter/arrayer overføres som referanse.]

  6. [Explain why a parameter is undefined when a function call is missing a parameter.]

    7. [Parameteren blir undefined fordi JavaScript setter manglende argumenter til undefined ved kall — parameteren er en lokal v ariabel som ikke får noen verdi hvis du ikke sender inn et argument (du kan bruke standardverdier for å unngå dette).]

  7. [Explain what happens when we use recursive functions.]

    8. [Rekursjon betyr at en funksjon kaller seg selv. Hver kall oppretter en ny stack‑frame med egne lokale variabler, og rekursjonen må ha en stopp‑tilstand (base case) ellers vil den fortsette til en stack overflow oppstår. Rekursive løsninger er naturlige for problemer som deles i mindre deler (f.eks. faktorielle, tre‑traversering), men kan være mindre effektive enn iterative løsninger hvis ikke optimalisert (f.eks. med memoization eller tail‑call).]
